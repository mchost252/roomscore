Here is the deep-dive technical architecture for image optimization, caching, and background synchronization.
I. The "Lightning" Image Pipeline
Proof images are the heaviest part of Krios. If you load 2MB images directly into a list, the app will stutter.
 * Triple-Tier Loading (The Progressive Blur):
   * Tier 1 (Instant): When a user uploads proof, generate a BlurHash (a 30-character string representing a blurred version of the image). Store this string in your database.
   * Tier 2 (Fast): Generate a Thumbnail (max 200px width) on the client before upload.
   * Tier 3 (Full): The high-res image only loads when a user taps the "Proof Card."
 * The "FastImage" Cache: Use react-native-fast-image. It performs aggressive disk caching. Once a user scrolls past a "Subway Node" with an image, it stays on their phone disk. It never hits the network again.
 * Auto-Purge (The Janitor): To prevent Krios from taking up 10GB of phone space, the app runs a "Janitor" function every midnight:
   * Scan the local media folder.
   * Delete any image file whose associated task_id is older than 5 days.
II. Caching & Persistence Architecture
You need a "Two-Speed" storage system to handle high-frequency UI updates and persistent data.
| Tool | Purpose | Data Type |
|---|---|---|
| MMKV | Ultra-fast Key/Value | UI State (Flip Header position, Theme, Last Sync Time). |
| SQLite | Relational Database | Messages, Tasks, Rooms, Member Lists. |
| Supabase | Cloud Sync | Global State, Authentication, Storage. |
The Delta-Sync Logic:
When the user opens a Room, the app asks: SELECT MAX(updated_at) FROM local_tasks WHERE room_id = x. It sends that single timestamp to Supabase. Supabase returns only the 3-4 things that changed since that second. This makes "loading" happen in under 100ms.

IV. Logic Calculations (The Performance Math)
1. The Doom Clock (GPU-Efficient)
Don't use a JS setInterval to move the clock; it will lag the UI. Instead, calculate the "Progress Percentage" once when the screen mounts and use react-native-reanimated to handle the visual transition.
2. The Ghost Approval Timer
Instead of a live countdown for every message, use a Static Target.
 * When a proof is rendered, calculate: const isApproved = Date.now() > (created_at + 14400000) (4 hours in ms).
 * If false, display the "Pending" shield.
 * If true and the status is still "Pending," the UI automatically shows the "Ghost Approved" shield locally while firing a background update to the server.
3. Weighted Aura (The Trust Shield)
To calculate a user's "Vote Weight" for a Vouch, use a simple multiplier:
 * Base Weight: 1.0

V. Background Sync (The "WhatsApp" Secret)
To ensure the user doesn't have to wait for "Connecting..." every time they open a room:
 * Supabase Realtime: Use a WebSocket connection while the app is in the foreground. When a "Node" is added to the Subway Line by someone else, it "pops" into the local SQLite DB and the UI updates instantly.
 * Background Tasks: Use Expo BackgroundFetch. Every 15 minutes, the app wakes up for 5 seconds, performs a Delta-Sync, and downloads new messages.
 * Silent Notifications: When an admin changes a setting, send a high-priority "Silent Push Notification" (FCM). The app receives it, updates the local database, and moves the folders around—all while the user's phone is in their pocket.
